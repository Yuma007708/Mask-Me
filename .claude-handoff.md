## 状況サマリ（2026-07-01 時点・最新）

### scanner 定期再生成の flaky 対策は不採用（悪化を確認）

コミット `f7834e5`（200 フレームごとに MediaPipe FaceLandmarker を作り直す）を
CI Run #28505454271 で検証した結果、**効果がないどころか悪化**した：

- 修正前（Run #28496967052）: `s1/off`・`s3/yunet`・`s5/off` の3ジョブのみ失敗
- 修正後（Run #28505454271）: 上記に加え **`s4/off`（38秒、これまで一度も失敗していない
  短尺動画）と `s5/yunet` も新たにクラッシュ**。ログを見ると `s4/off` はフレームループ
  終盤で無言でテストプロセスが再起動（`Restarting after unexpected exit, crash, or test
  timeout`）→ 0 tests executed で `TEST FAILED`（exit code 65）。
- 旧コード（scanner 使い回し）では `s4/off` は問題なく通っていた実績があるため、
  **scanner を実行中に作り直すこと自体が Simulator 上で新しい不安定要因になっている**
  と判断し、この対策は revert（コミット `dbe81d2`）。

### 新たに判明した別の失敗モード：ジョブタイムアウト

同じ Run で `s1/yunet`・`s1/faceDetector`・`s3/off` の3ジョブが `cancelled` に
なっていたが、これは **クラッシュではなく `timeout-minutes: 60` にほぼ正確に
到達したことによるタイムアウト**（60:32〜60:41 で揃って切れている）。
`s1/faceDetector` は「安定している」とされてきた backend だったため、これは
クラッシュ系の flaky とは別に、**長尺動画の処理自体が1時間を超えかねないほど遅い
（またはランナーが混雑している）** という新しい問題。

### 対策：動画を前半/後半に分割して別プロセスで実行

コミット `dbe81d2` で対応：

- scanner 定期再生成を revert し、旧来通り 1 テストにつき 1 scanner インスタンスに戻した
- 各動画・各 backend のテストを **前半 (A) ・後半 (B) の2つの XCTest メソッドに分割**
  （`test_S1_off_A` / `test_S1_off_B` など、計 30 メソッド）。CI matrix にも `half: [A, B]`
  軸を追加し、`.github/workflows/dvalid.yml` の Aggregate ステップで A+B を合算して元の
  「動画全体」相当の rate に戻して表示する
- 狙い: 1 プロセスあたりの累積 detect 呼び出し数・実行時間を半分にすることで、クラッシュ
  条件（ネイティブリソース蓄積と推測）にもジョブタイムアウトにも到達しにくくする
- ローカルで `xcodegen generate` → `pod install` → `xcodebuild build-for-testing` /
  `-only-testing:MaskMeTests/DValidVideoTests/test_S1_off_A`（動画無し・XCTSkip）まで確認済み。
  実動画での検証は CI 待ち。
### 前後半分割の検証結果（Run #28523026472）と追加対策：CI リトライ

Run #28523026472（`videos=s1,s3,s4,s5` `backends=off,faceDetector,yunet` `halves=A,B` の
24ジョブ、s2は安定実績があるため除外）の結果:

- **timeout-minutes によるキャンセルは完全に解消**（前回3件→0件）。前後半分割で
  1ジョブあたりの実行時間が短くなったことが効いている
- **クラッシュ失敗率は改善したが解消はしなかった**: 24ジョブ中5件失敗
  （`s1/yunet/B`・`s1/faceDetector/A`・`s3/yunet/A`・`s4/faceDetector/B`・`s5/faceDetector/B`）。
  分割前は15ジョブ中8件が失敗/cancelledだったので、率としては約53%→21%に改善
- **重要な発見**: 失敗した5件のログを確認したところ、全て同じ
  `Restarting after unexpected exit, crash, or test timeout` → `Executed 0 tests` →
  `TEST FAILED` (exit 65) のパターン。**これまで一度もクラッシュしたことがなかった
  `.faceDetector` backend でも今回3件クラッシュしている**（s1/A, s4/B, s5/B）。
  つまりこの crash は「長尺動画 + off/yunet」に固有ではなく、**backend や動画長を問わず
  確率的に起きるプロセスクラッシュ**である可能性が高い。クラッシュレポート（.ips や
  xcresult 内の diagnostic）は workflow 側で一切収集していないため、スタックトレース
  ベースの原因究明はまだできていない

この状況（3系統すべてのbackendで、動画長に関係なく確率的にクラッシュ）から、
「MediaPipe VIDEO モードの内部状態蓄積」という当初の仮説だけでは説明がつかなくなり、
CI ランナー（macOS Simulator）側のインフラ起因の flaky である可能性を重視することにした。

**対策（コミット `cc064cb`）**: `.github/workflows/dvalid.yml` の `Run XCTest` ステップで
`xcodebuild test` が失敗したら **最大3回まで自動リトライ**するようにした
（`until ... do ... done` ループ、`PIPESTATUS[0]` で xcodebuild 自体の exit code を見る）。
前後半分割で失敗率を下げた上に、残った確率的クラッシュはリトライで吸収する狙い。

- **次の一手**: リトライ込みで再検証中。前回失敗した5組み合わせ
  （s1/yunet, s1/faceDetector, s3/yunet, s4/faceDetector, s5/faceDetector の該当half）
  を中心に絞ったCI runで様子を見る。リトライでも解消しない/リトライ自体が頻発するようなら、
  workflow に xcresult からのクラッシュログ抽出ステップを追加して実際のクラッシュ原因
  （OOM か シグナルか）を特定するフェーズに進む。

**結論**:

- 3 backend (off/faceDetector/yunet) は S1〜S4 で hit 完全一致、S5 でだけ yunet が +3 件追加（letterbox の効果が S5 特定シーンで現れる）

## flaky パターン

s1/off, s1/yunet, s3/off, s3/yunet は連続 rerun でも再失敗。`.faceDetector` は通る。
- 動画長 92秒以上 + 補助検出器が MP FaceLandmarker 共有以外（`.off`=なし, `.yunet`=YuNet）
- テストプロセスがループ末尾近辺（≒ 全フレーム処理完了直前）でクラッシュ → 0 tests executed
- 短尺 (s2=29s, s4=38s) や `.faceDetector` バックエンドでは発生せず
- s5 (104s) は時々発生

**今後の対策候補**:
1. テスト内の scanner を 200 フレームごとに再作成（MediaPipe キャッシュ抑制）
2. interval 1/15 → 1/30 で処理量半減（精度比較として弱くなる）
3. 1 テスト = 1 動画半分（前後分割で 2 テスト化）
